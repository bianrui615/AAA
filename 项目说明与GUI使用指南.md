# 北斗定位解算全流程软件系统说明与 GUI 使用指南

## 1. 项目目的

本项目面向 `beidou_codex_requirements.md` 中整理的“北斗定位解算全流程软件系统”基础任务，目标是围绕一份 RINEX NAV 导航文件，完成从广播星历解析、卫星位置与钟差计算、伪距模拟、单点定位解算，到连续历元定位分析和 GUI 可视化展示的完整流程。

需要特别说明的是，题目原始要求中提到同时分析 OBS 观测文件和 NAV 导航文件；但本项目当前代码方案不读取真实 OBS 文件，而是采用“读取 NAV 广播星历 + 给定真实接收机坐标 + 生成模拟伪距”的方式完成 SPP 解算验证。因此本文档后续所有“输入/输出/完成要求”的说明，都以当前代码实现为准：重点说明 NAV 解析、卫星位置与钟差计算、模拟伪距、SPP、连续定位、GUI 和测试报告，不展开 OBS 观测文件解析流程。

另外，`beidou_codex_requirements.md` 中的 AI 定位误差预测与补偿属于可选提高部分，本文档暂不纳入完成范围。

系统当前的定位精度检测方式是：给定一个接收机真实坐标，系统根据广播星历计算卫星位置，生成带误差项的模拟伪距，再用 SPP 单点定位算法反算接收机坐标，最后比较“反算坐标”和“真实坐标”的三维距离误差。

换句话说，本项目验证的是：

- RINEX NAV 解析是否正确；
- 广播星历卫星位置计算是否合理；
- 健康卫星筛选是否生效；
- 模拟伪距模型是否能驱动 SPP 解算；
- 单历元和连续历元定位结果是否能稳定输出；
- GUI 是否能完成参数输入、运行控制、结果展示和图表查看。

## 2. 整体解题思路

### 2.1 与题目基础要求的对应关系

题目文件中的基础要求可以概括为：RINEX 数据解析、卫星位置与钟差计算、SPP 单点定位、连续定位与精度分析、GUI 交互和测试报告。本项目按当前代码实现完成情况如下：

| 题目基础要求 | 当前项目实现方式 |
| --- | --- |
| RINEX 数据解析与预处理 | 已实现 RINEX NAV 导航文件解析、北斗卫星筛选、健康状态读取与检查；未读取 OBS 文件，观测伪距由模块三模拟生成 |
| 卫星位置、钟差与传播误差处理 | 已根据广播星历手写计算卫星 ECEF 坐标、卫星钟差、相对论改正，并进行健康状态筛选；电离层/对流层影响以模拟误差项进入伪距模型 |
| 伪距单点定位 SPP | 已实现伪距观测方程、迭代最小二乘、至少 4 颗卫星检查、接收机钟差估计、PDOP/GDOP 和 ECEF 转经纬高 |
| 连续定位与结果分析 | 已实现多历元循环解算、逐历元结果保存、平均误差、RMS、最大/最小误差、卫星数和 DOP 统计 |
| 可视化 | 已输出误差曲线、经纬度轨迹图、卫星数量与 DOP 曲线，并在 GUI 中显示表格、轨迹回放和误差曲线 |
| 系统整合、GUI 与测试报告 | 已实现命令行总流程、GUI 参数输入与运行、模块输出文件、系统测试报告 |
| 可选 AI 提高部分 | 暂不考虑，当前项目未实现机器学习误差预测与补偿 |

整个模型按照 GNSS 定位的典型数据流设计：

1. 读取 RINEX NAV 文件，解析北斗卫星广播星历。
2. 在指定历元，为每颗卫星选择最接近当前时刻且健康状态正常的星历。
3. 根据广播星历参数计算卫星在 ECEF 坐标系下的位置和卫星钟差。
4. 给定接收机真实位置，计算卫星到接收机的几何距离，并叠加模拟误差生成伪距。
5. 使用伪距观测方程，通过迭代最小二乘法估计接收机 ECEF 坐标和接收机钟差。
6. 将解算坐标转换为经纬高，并与真实坐标比较，得到三维定位误差。
7. 对多个历元重复上述过程，统计平均误差、RMS 误差、最大误差、最小误差、PDOP/GDOP 等指标。
8. 通过 GUI 提供 NAV 文件导入、参数设置、真实坐标输入、结果表格、误差曲线和轨迹回放。

核心坐标系说明：

- ECEF：地心地固直角坐标系，单位为 m，坐标形式为 `X, Y, Z`。
- 经纬高：WGS84/CGCS2000 大地坐标，纬度和经度单位为 deg，高度单位为 m。
- GUI 支持两种真实坐标输入方式：直接输入 ECEF，或者输入经纬高后自动转换为 ECEF。

## 3. 模块一：RINEX NAV 导航文件解析

文件：`module1_nav_parser.py`

### 3.1 模块作用

模块一负责读取 RINEX NAV 导航文件，解析其中的北斗广播星历记录。程序只保留卫星编号以 `C` 开头的北斗卫星，例如 `C01`、`C02`。解析结果会被后续模块用于卫星位置和钟差计算。

### 3.2 输入内容

输入是一份 RINEX NAV 文件，例如：

```text
tarc0910.26b
```

文件中需要包含 RINEX 文件头、`END OF HEADER` 标记，以及若干 8 行一组的导航记录。

### 3.3 处理逻辑

模块一主要完成以下工作：

- 读取文件内容并识别 RINEX 版本；
- 跳过文件头，从 `END OF HEADER` 后开始读取导航记录；
- 每 8 行解析一颗卫星的一个广播星历记录；
- 只保留北斗卫星记录；
- 兼容 `D` 和 `E` 两种科学计数法；
- 提取卫星钟差参数、轨道参数、健康状态、TGD 参数等；
- 按卫星编号整理为字典结构；
- 对每颗卫星的星历按 `toc` 时间排序；
- 统计解析成功、跳过、失败、不完整记录数量；
- 检查健康状态，标记 `health != 0` 的异常星历。

### 3.4 输出内容

模块一输出两个文件：

```text
output/module1_nav_parse_summary.txt
output/module1_ephemeris_list.csv
```

`module1_nav_parse_summary.txt` 表示解析摘要，主要字段含义：

- `NAV 文件路径`：本次解析使用的导航文件；
- `RINEX 文件版本`：文件头中的 RINEX 版本；
- `成功解析的北斗卫星数量`：解析到的北斗卫星颗数；
- `成功解析的广播星历记录总数`：所有北斗星历记录总数；
- `跳过的非北斗记录数`：非北斗卫星记录数量；
- `不完整记录数`：不足 8 行的异常记录数量；
- `解析失败的北斗记录数`：字段无法解析的记录数量；
- `每颗北斗卫星对应的星历记录数`：各卫星可用星历数量；
- `健康状态检查`：列出 health 非 0 的异常卫星和异常记录数量。

`module1_ephemeris_list.csv` 表示星历明细，主要字段含义：

- `sat_id`：卫星编号；
- `toc_time`：卫星钟参考时间；
- `toe`：星历参考时刻，单位为周内秒；
- `af0, af1, af2`：卫星钟差多项式参数；
- `sqrtA`：轨道长半轴平方根；
- `e`：轨道偏心率；
- `i0`：轨道倾角；
- `Omega0`：升交点赤经；
- `omega`：近地点幅角；
- `M0`：参考时刻平近点角；
- `delta_n`：平均角速度改正量；
- `health`：卫星健康状态，通常 `0` 表示健康。

### 3.5 对应题目基础要求

题目基础要求中包含 RINEX OBS 和 NAV 两类文件解析。按照当前代码方案，模块一完成的是 NAV 导航文件部分：读取 RINEX NAV、提取北斗广播星历参数、读取卫星健康状态、输出解析结果和统计信息，并为后续卫星位置计算提供基础数据。OBS 观测文件解析部分当前不实现，由模块三的模拟伪距方案替代真实观测伪距输入。

## 4. 模块二：卫星位置与钟差计算

文件：`module2_satellite_position_clock.py`

### 4.1 模块作用

模块二根据模块一解析出的广播星历，在指定测试历元计算每颗健康北斗卫星的 ECEF 坐标和卫星钟差。

### 4.2 输入内容

输入包括：

- 模块一输出的星历数据 `nav_data`；
- 一个测试历元时间，例如 `2026-04-01 00:00:00`；
- 每颗卫星的健康状态 `health`；
- 广播星历中的轨道参数和钟差参数。

### 4.3 处理逻辑

模块二主要完成以下工作：

- 对每颗卫星选择与当前历元最接近的健康星历；
- 若没有健康星历，则跳过该卫星；
- 根据广播星历计算轨道长半轴、平均角速度、平近点角；
- 迭代求解 Kepler 方程，得到偏近点角；
- 计算真近点角、升交距角、摄动改正项；
- 计算轨道平面坐标；
- 将轨道平面坐标转换为 ECEF 坐标；
- 对 GEO 卫星进行特殊倾角旋转处理；
- 计算相对论效应改正；
- 计算卫星钟差；
- 检查卫星位置模长是否在合理数量级内；
- 将健康状态写入输出文件，保证后续模块只使用健康卫星。

### 4.4 输出内容

模块二输出两个文件：

```text
output/module2_satellite_position_clock.csv
output/module2_satellite_position_summary.txt
```

`module2_satellite_position_clock.csv` 表示每颗卫星在测试历元的计算结果，字段含义：

- `epoch_time`：测试历元；
- `sat_id`：卫星编号；
- `X, Y, Z`：卫星 ECEF 坐标，单位为 m；
- `satellite_clock_bias`：卫星钟差，单位为 s；
- `relativistic_correction`：相对论效应改正，单位为 s；
- `position_norm`：卫星位置向量模长，单位为 m；
- `health`：卫星健康状态；
- `status`：计算状态，例如 `计算成功`、`跳过：未找到健康星历`、`卫星坐标数量级异常`。

`module2_satellite_position_summary.txt` 表示卫星位置计算摘要，主要说明：

- 候选北斗卫星数量；
- 参与计算的健康卫星数量；
- 因健康状态异常或无健康星历跳过的卫星数量；
- 坐标数量级检查结果；
- 卫星钟差范围。

### 4.5 对应题目基础要求

模块二完成了题目中“根据广播星历计算卫星位置”“计算卫星钟差”“输出卫星中间结果”“检查结果合理性”的基础要求。当前实现包含相对论改正和健康状态筛选；电离层、对流层传播影响没有按真实 OBS 观测值做改正，而是在模块三生成模拟伪距时作为误差项加入。

## 5. 模块三：模拟伪距与单历元 SPP 解算

文件：`module3_spp_solver.py`

### 5.1 模块作用

模块三负责在单个历元下生成模拟伪距，并使用迭代最小二乘方法完成 SPP 单点定位解算。

### 5.2 输入内容

输入包括：

- 模块二计算得到的健康卫星 ECEF 坐标；
- 卫星健康状态；
- 接收机真实坐标；
- 测试历元时间；
- 随机种子；
- SPP 最大迭代次数和收敛阈值。

接收机真实坐标可以是：

- 代码或 GUI 中直接输入的 ECEF 坐标；
- GUI 中输入的经纬高坐标自动转换得到的 ECEF 坐标。

### 5.3 处理逻辑

模块三主要完成以下工作：

- 再次根据 `health=0` 筛选健康卫星；
- 计算卫星到真实接收机位置的几何距离 `rho`；
- 同一历元内生成一个公共接收机钟差，并对所有卫星共用；
- 使用模块二给出的卫星钟差生成原始伪距，并进行卫星钟差改正；
- 使用简化高度角电离层模型和简化 Saastamoinen 对流层模型生成并改正传播延迟；
- 叠加 SISRE 和观测噪声，生成用于 SPP 的改正后模拟伪距；
- 构建伪距观测方程；
- 使用迭代最小二乘法求解接收机位置改正量和接收机钟差；
- 判断是否收敛；
- 将解算得到的 ECEF 坐标转换为经纬高；
- 计算 PDOP 和 GDOP；
- 与真实坐标比较，得到单历元三维误差。

当前模拟伪距模型包括：

- SISRE 误差；
- 简化电离层延迟；
- 简化 Saastamoinen 对流层延迟；
- 同历元公共接收机钟差；
- 模块二卫星钟差改正；
- 观测噪声。

### 5.4 输出内容

模块三输出两个文件：

```text
output/module3_pseudorange_single_epoch.csv
output/module3_spp_result_single_epoch.txt
```

`module3_pseudorange_single_epoch.csv` 表示伪距明细，字段含义：

- `epoch_time`：历元时间；
- `sat_id`：卫星编号；
- `sat_X, sat_Y, sat_Z`：卫星 ECEF 坐标；
- `health`：卫星健康状态；
- `satellite_clock_bias`：模块二计算得到的卫星钟差，单位为 s；
- `satellite_clock_correction_m`：卫星钟差对应的距离改正量，单位为 m；
- `elevation_deg`：卫星相对接收机的高度角；
- `true_receiver_X, true_receiver_Y, true_receiver_Z`：真实接收机 ECEF 坐标；
- `rho`：卫星到接收机的几何距离；
- `sisre_error`：卫星信号空间误差模拟项；
- `ionosphere_error`：简化电离层延迟项；
- `troposphere_error`：简化 Saastamoinen 对流层延迟项；
- `receiver_clock_error`：接收机钟差模拟项；
- `noise_error`：观测噪声；
- `raw_simulated_pseudorange`：包含卫星钟差、传播延迟和噪声的原始模拟伪距；
- `simulated_pseudorange`：应用卫星钟差、电离层和对流层改正后的伪距，也是 SPP 解算实际使用的伪距。

`module3_spp_result_single_epoch.txt` 表示单历元 SPP 报告，字段含义：

- `测试历元时间`：本次单历元解算时间；
- `参与定位的卫星数量`：进入 SPP 解算的健康卫星数量；
- `健康状态筛选`：确认仅使用 `health=0` 的卫星；
- `接收机真实 ECEF 坐标`：用于精度检测的真值；
- `解算得到的接收机 ECEF 坐标`：SPP 反算结果；
- `解算得到的经纬高坐标`：将 ECEF 转换后的可读位置；
- `接收机钟差估计值`：以 m 表示的接收机钟差项；
- `三维定位误差`：解算坐标与真实坐标之间的距离；
- `PDOP, GDOP`：卫星几何精度因子；
- `迭代次数`：最小二乘迭代次数；
- `是否收敛`：解算是否满足收敛阈值；
- `解算状态`：成功或失败；
- `失败原因`：失败时给出原因。

### 5.5 对应题目基础要求

模块三完成了题目中“建立伪距定位方程”“实现迭代最小二乘 SPP”“输出接收机坐标、钟差、DOP 和误差”的基础要求。由于当前项目不读取 OBS 文件，伪距观测量由系统根据卫星位置、真实接收机坐标和误差模型模拟生成，而不是来自真实观测文件。

## 6. 模块四：连续定位与结果分析

文件：`module4_continuous_analysis.py`

### 6.1 模块作用

模块四将模块二和模块三扩展到多个历元，完成连续定位仿真、误差统计和图表绘制。

### 6.2 输入内容

输入包括：

- 模块一解析得到的 `nav_data`；
- 连续仿真的开始时间；
- 连续仿真的结束时间；
- 采样间隔；
- 接收机真实 ECEF 坐标；
- 随机种子；
- SPP 最大迭代次数；
- 收敛阈值；
- 输出目录。

### 6.3 处理逻辑

模块四主要完成以下工作：

- 从开始时间到结束时间按采样间隔生成多个历元；
- 每个历元选择健康星历；
- 每个历元计算卫星位置；
- 每个历元生成模拟伪距；
- 每个历元调用 SPP 解算；
- 上一历元成功解算坐标会作为下一历元初值；
- 记录每个历元的状态、卫星数、坐标、DOP、误差；
- 统计成功率、平均误差、RMS 误差、最大误差、最小误差；
- 绘制误差曲线、轨迹图、卫星数量与 DOP 曲线。

### 6.4 输出内容

模块四输出：

```text
output/module4_continuous_position_results.csv
output/module4_error_statistics.txt
output/module4_error_curve.png
output/module4_trajectory.png
output/module4_satellite_dop_curve.png
```

`module4_continuous_position_results.csv` 字段含义：

- `epoch_time`：历元时间；
- `status`：该历元解算状态；
- `satellite_count`：可用健康卫星数量；
- `X, Y, Z`：解算得到的接收机 ECEF 坐标；
- `lat, lon, height`：解算得到的经纬高；
- `clock_bias`：接收机钟差估计值，单位为 m；
- `PDOP, GDOP`：精度因子；
- `error_3d`：三维定位误差，单位为 m；
- `iteration_count`：迭代次数；
- `failure_reason`：失败原因。

`module4_error_statistics.txt` 字段含义：

- `总历元数`：参与连续定位的历元数量；
- `成功解算历元数`：成功收敛的历元数量；
- `失败历元数`：失败历元数量；
- `成功率`：成功历元占比；
- `平均可用卫星数量`：每个历元平均可用卫星数；

### 8.3 GUI 的两种测试方法

GUI 的“接收机真实坐标”区域提供两个页签。

#### 方法一：ECEF

适用于已经知道接收机 ECEF 坐标的情况。

填写：

```text
X(m)
Y(m)
Z(m)
```

例如默认值：

```text
X = -2267800.0
Y = 5009340.0
Z = 3221000.0
```

这个默认坐标换算成经纬高约为：

```text
纬度 = 30.5285233051°
经度 = 114.3569441195°
高度 = 41.8796 m
```

点击 `开始解算` 后，系统直接把该 ECEF 坐标作为真实接收机位置进行误差检测。

#### 方法二：经纬高

适用于只知道经纬度和高度的情况。

填写：

```text
纬度(deg)
经度(deg)
高度(m)
```

填写规则：

- 北纬为正数，南纬为负数；
- 东经为正数，西经为负数；
- 不要输入 `N`、`S`、`E`、`W`、`°` 等字符，只输入数字；
- 高度单位为 m；
- 坐标建议使用 WGS84 或 CGCS2000。

示例：北京邮电大学沙河校区实验坐标：

```text
纬度：40.1575
经度：116.2885
高度：35
```

因为它是北纬、东经，所以纬度和经度都填正数。

再例如：

```text
南纬 33.9°，西经 151.2°，高度 20 m
```

应填写：

```text
纬度：-33.9
经度：-151.2
高度：20
```

GUI 会自动显示换算后的 ECEF：

```text
自动换算：X=... m，Y=... m，Z=... m
```

点击 `转换并填入 ECEF` 可以把经纬高换算结果同步到“方法一：ECEF”页签。即使不点击该按钮，只要当前页签停留在“方法二：经纬高”，点击 `开始解算` 时也会自动转换并用于检测。

### 8.4 GUI 运行逻辑

点击 `开始解算` 后，GUI 执行以下流程：

1. 检查是否已导入有效 NAV 文件；
2. 读取起始时间、结束时间、采样间隔、迭代参数；

4. 若选择经纬高页签，则自动转换为 ECEF；

6. 每完成一个历元就向表格追加一行结果；

9. 弹窗提示完成。

## 9. 常见注意事项

1. NAV 文件必须覆盖所选时间范围附近，否则可能没有可用星历。
2. 参与 SPP 至少需要 4 颗健康卫星。
3. `health != 0` 的卫星会被筛除，不参与定位。
4. 经纬高输入时，北纬和东经为正，南纬和西经为负。
5. 高度应尽量使用椭球高；如果使用普通海拔，可能引入一定高度偏差。
6. 当前系统使用模拟伪距，不读取真实 OBS 观测伪距，因此误差反映的是仿真模型和解算流程的表现。
7. `beidou_codex_requirements.md` 中的 OBS 解析、真实伪距改正和多组实测数据测试属于原始题目完整方向；本项目当前按代码实现方案，将 OBS 部分替换为模拟伪距输入。
8. AI 定位误差预测与补偿属于提高部分，本文档和当前项目暂不考虑。
9. 若要验证真实接收机精度，需要后续增加 OBS 观测文件解析、真实伪距读取、传播延迟改正和实测参考坐标对比模块。
